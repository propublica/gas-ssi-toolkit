# AI Column Visibility Markers — Design Spec

**Date:** 2026-05-22
**Status:** Approved

## Problem

AI-generated cell content is indistinguishable from human-entered data in Google Sheets. Users may treat Gemini output as authoritative without verifying it. The `ai_*` column name prefix is the only current signal, and it only applies to newly created columns.

## Goals

- Make AI-generated output visually obvious at a glance
- Communicate to users that content should be verified before use
- Prevent accidental edits to cells being actively written by a running job

## Non-goals

- Marking the `_grounding` column (grounding values are deterministic API returns, not probabilistic AI output)
- Retroactively marking columns from past runs
- Blocking edits after the run completes

---

## Design

Two complementary mechanisms are added to `runBatchAI`, applied per-chunk:

### 1. Temporary Protection (process guard)

At the start of each `runBatchAI` invocation, a range protection is applied to the output column cells that are about to be written. This signals to users that the column is actively being processed.

- **Range:** header cell (row 1) + data rows `startRow` through `startRow + numRows - 1`
- **Description:** `"AI run in progress — please wait"`
- **Lifecycle:** added before the write phase, removed in a `finally` block (guaranteed cleanup even on error)

Because `runBatchAI` is called sequentially per chunk by the client (`runChunks` in `configure-ai-run.ts`), the protection covers only the current chunk's rows at any moment. By the time chunk 2 starts, chunk 1's rows are already unlocked and amber-marked.

### 2. Permanent Amber Marking (epistemic marker)

After `SpreadsheetApp.flush()` completes successfully, two formatting calls mark the output:

- **Header cell:** amber background (`#F9AB00`) + note `"AI-generated by Gemini — verify before publishing"`
- **Data cells:** light amber wash (`#FFF8E1`) on `startRow` through `startRow + numRows - 1`

The header note is idempotent — re-applied on every chunk, but the content never changes. The amber wash spreads down the column chunk by chunk as the run progresses, giving users a visible progress signal.

Amber is applied inside the `try` block, so it only lands if the write phase completes without throwing.

---

## Execution Flow (per chunk)

```
protectAIOutputRange(sheet, outputIdx, startRow, numRows)
try {
  [existing wave 1 + wave 2 write logic]
  SpreadsheetApp.flush()
  markAIOutputRange(sheet, outputIdx, startRow, numRows)
  toast
} finally {
  protection.remove()
}
```

## Implementation

### New helpers in `src/server/utils.ts`

**`protectAIOutputRange`**

```ts
protectAIOutputRange(
  sheet: GoogleAppsScript.Spreadsheet.Sheet,
  colIdx: number,   // 1-based
  startRow: number,
  numRows: number,
): GoogleAppsScript.Spreadsheet.Protection[]
```

Protects header cell + data rows for the output column. Returns an array of `Protection` objects (one for the header cell, one for the data range — they are non-contiguous when `startRow > 2`) so the caller can call `.remove()` on each in `finally`.

**`markAIOutputRange`**

```ts
markAIOutputRange(
  sheet: GoogleAppsScript.Spreadsheet.Sheet,
  colIdx: number,   // 1-based
  startRow: number,
  numRows: number,
): void
```

Applies amber background to header + data cells; sets note on header cell. Two range calls total.

### Changes in `src/server/index.ts`

`runBatchAI` is updated to:

1. Call `protectAIOutputRange` immediately after resolving `outputIdx`
2. Wrap the existing write loop + flush in `try/finally`
3. Call `markAIOutputRange` after `SpreadsheetApp.flush()`, inside the `try`
4. Call `.remove()` on each protection in `finally`

No changes to the function signature, `RunConfig`, shared types, client code, or `rollup.config.js`.

## Colours

| Token | Hex | Use |
|---|---|---|
| Amber header | `#F9AB00` | Header cell background — bold, immediately visible |
| Amber wash | `#FFF8E1` | Data cell background — subtle, doesn't impede reading |

## Tests

`protectAIOutputRange` and `markAIOutputRange` accept `sheet` as a parameter, so they follow the existing duck-typed fake pattern in `__tests__/`. Tests verify the correct ranges and values are passed to the GAS mock — `setBackground`, `setNote`, and `protect` calls with expected arguments.

No changes to existing tests. New test file: `__tests__/utils-ai-marking.test.ts` (or added to `__tests__/utils.test.ts` if the file stays manageable).
